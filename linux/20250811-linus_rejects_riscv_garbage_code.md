# [Linus怒斥“垃圾代码”：一次合并请求，为何引发对开源未来的深思？](20250811-linus_rejects_riscv_garbage_code.mp3)

近日，Linux内核社区再次因Linus Torvalds的尖锐言论而波澜四起。[他正式拒绝了原计划并入Linux 6.17内核的RISC-V架构更新包](https://www.phoronix.com/news/Linux-6.17-RISC-V-Rejected)，其理由简单而直接：提交太晚，且部分代码质量堪称“垃圾”。这一看似平常的开发流程事件，却如同一面棱镜，折射出顶级开源项目内部复杂的文化冲突与治理困境。

## 导火索：一封迟到且“画蛇添足”的合并请求

事件的直接原因是RISC-V维护者在合并窗口即将关闭的周五才提交了拉取请求（Pull Request），这严重违反了Linus因个人差旅安排而提出的“尽早提交”的要求。然而，真正点燃Linus怒火的，是代码本身的质量问题。

Linus将矛头直指一个名为 `make_u32_from_two_u16(a,b)` 的辅助函数。这个函数被添加到了一个通用的、非RISC-V专属的头文件中。Linus对此评论道：

> “不。这是垃圾，而且来得太晚了……它让世界变得更糟。如果你直接写成 `(a << 16) + b`，每个人都清楚知道它的作用和字节顺序。相反，用了这个愚蠢的‘辅助函数’，你根本不知道哪个是高位，哪个是低位……你把事情搞得更糟了。”

在Linus看来，这个函数不仅毫无必要，反而降低了代码的可读性和可维护性，是一种典型的“画蛇添足”。将其放入通用头文件的行为，更是污染了公共代码空间，是绝对不可接受的。因此，他毫不犹豫地驳回了整个请求，要求其在下一个版本（6.18）的合并窗口早期，剔除“垃圾”后重新提交。

## “暴君”还是“质检员”？社区对Linus风格的撕裂

Linus的严厉措辞立刻在各大技术社区引发热议，并形成了鲜明的两大阵营。

**支持者认为，Linus的“刻薄”是Linux成功的关键。** 他们坚信，对于Linux这样一个支撑着全球无数关键系统的庞大项目而言，任何微小的代码瑕疵都可能引发灾难性后果。[Linus作为最终的“守门人”](https://www.reddit.com/r/RISCV/comments/1mlqcb0/linus_torvalds_rejects_riscv_changes_for_linux/)，其不留情面的直接批评，是确保代码质量、抵制平庸的必要手段。在他们看来，开发者应该有“更厚的脸皮”，专注于技术改进，而非纠结于沟通的语气。

**然而，另一方则对此表示强烈反对。** 他们认为[Linus的言语暴力构成了职场霸凌](https://www.phoronix.com/forums/forum/phoronix/latest-phoronix-articles/1567875-linus-torvalds-rejects-risc-v-changes-for-linux-6-17-garbage)，公然违反了Linux项目自己制定的《行为准则》（Code of Conduct）。这种“有毒”的沟通方式会吓跑许多有才华但性格更温和的工程师，长远来看对社区生态是一种损害。正如一位[Hacker News](https://news.ycombinator.com/item?id=44853086)用户所言：“Linus就是Linus，但有时候，如果他能不那么‘Linus’一点，世界会更美好。”

## “自家后院的垃圾”：双标争议与内核治理的深层忧虑

风波中，Linus的一句话引发了更深层次的争议：“我希望RISC-V部分内部没有垃圾，但那是你们的选择。”

批评者将此解读为一种“双重标准”：只要别污染我的“公共地盘”，你在自家“后院”里堆点垃圾我无所谓。这引发了对内核整体完整性的担忧。如果允许某些子系统存在低质量代码，那么整个Linux的质量神话将如何维系？这也被拿来与悬而未决的bcachefs文件系统争议做比较，指责Linus在规则执行上存在不一致。

更重要的是，这场风波再次将一个老问题推到台前：**后Linus时代的Linux将何去何从？**

社区普遍担忧，当这位拥有绝对权威和独特个性的“终身仁慈独裁者”退休后，谁能接替他来维持如此之高的代码标准？一个由委员会或多位维护者共同管理的模式，是否还有魄力对重要的贡献者说“不”，并斥责其工作是“垃圾”？许多人悲观地认为，届时，出于人情世故和流程主义，内核的质量可能会不可避免地滑坡。

从一次代码合并被拒，到对整个开源生态未来的拷问，Linus的这一次“发飙”，无疑为我们提供了一个观察顶级技术项目内部运作逻辑的绝佳窗口。
