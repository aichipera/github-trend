# All Aboard 系列，第 7 部分：在 RISC-V 上进入和退出 Linux 内核

> 本文于**2017年10月23日**发布于 https://www.sifive.com/blog/all-aboard-part-7-entering-and-exiting-the-linux-kernel-on-risc-v

继续我们的 RISC-V Linux 内核移植之旅，本周我们将讨论上下文切换。上下文切换是架构移植中较为重要的部分之一：完全抽象掉进入和退出内核的细节几乎是不可能的。由于它处于许多关键路径上（系统调用和调度），它必须快速；但由于它是内核与用户空间之间的唯一一道防线，它也必须安全。

## RISC-V 系统上的陷阱 (Traps)

RISC-V ISA 一个更有趣的地方是，在发生中断或异常时，硬件做的事情非常少。这除了使 ISA 易于实现外，还有一个优点，即允许软件在一个干净的石板上实现上下文切换。

RISC-V 监督者规范定义了一个单一的内核陷阱入口点，可以通过写入 `stvec` CSR 来设置。将控制权转移到内核的唯一方法就是通过这个入口点，而陷入陷阱的唯一副作用是改变 PC、异常 PC（`sepc`）和异常原因（`scause`）CSR，以及特权模式。监督者软件被期望提供用户空间 ABI 的透明实现。

就像通过单一陷阱入口点进入内核一样，离开内核的唯一方法是执行 `sret` 指令。这与陷入陷阱是镜像操作：所有发生的事情就是特权模式被改变，PC 被重置为异常 PC CSR 的值。同样，监督者软件被期望提供用户空间 ABI 的透明实现。

RISC-V ISA 中对监督者模式陷阱处理的另一个支持是 `sscratch` CSR。这个 CSR 提供了一个大小为 XLEN 的保存区域，它没有隐式行为，因此完全由监督者自行决定如何使用。我们所有的软件上下文切换实现都使用这个寄存器来存储一个指向内存区域的指针，该内存区域包含实际进行上下文切换所需的任何额外信息，基本上就是将整个上下文切换的实现推到了软件层面。

### 陷阱委托 (Trap Delegation)

RISC-V ISA 定义了默认情况下所有陷阱都由机器模式处理，但可以选择将某些特权级别的陷阱直接委托给它们各自的特权级别处理。监督者模式软件对用于进入其陷阱处理程序的机制是无感的，它只是假设相关的陷阱最终会到达监督者模式。

机器模式软件被期望在硬件或软件中过滤陷阱。例如，如果机器模式实现对某些不支持的指令有仿真程序，它将不得不在软件中处理非法指令陷阱，但可以通过硬件机制委托其余的陷阱。

陷阱委托机制通过将陷阱直接委托给监督者模式硬件来支持高性能陷阱，同时仍然允许在较低特权模式中处理任何陷阱的灵活性。这允许简单地实现虚拟机，同时在不使用虚拟化的实现上仍能保持高性能。

### `handle_exception`，陷阱入口点

由于 RISC-V 系统上上下文切换的大部分有趣方面都由监督者模式软件处理，硬件通过一个单一的陷阱入口点进入内核，然后由监督者模式软件决定如何处理该陷阱。RISC-V ISA 定义了两类陷阱：

*   中断（Interrupts），是异步的。RISC-V 定义了软件中断、定时器中断和外部中断。
*   异常（Exceptions），是同步的。RISC-V 定义了处理指令、加载、存储和 AMO 访问故障的异常；环境调用（在 Linux 上用于系统调用）；非法指令；和断点。

陷阱类型是在进入陷阱处理程序时由 `scause` CSR 决定的。在将整数寄存器保存到内核堆栈（可以通过 `sscratch` 查找到）之后，我们检查陷阱原因并决定如何处理该陷阱。RISC-V 通过在 `scause` 中设置高位来区分中断，这使得很容易过滤掉并处理它们。由于大多数异常是由用户空间发出 `scall` 指令以开始一个系统调用而产生的，我们接着检查这种情况并使用 Linux 的通用系统调用处理基础设施来处理系统调用。其余的异常通过一个跳转表来处理，每个都有一个相当直接的实现，将控制权交还给内核相关的通用异常处理例程。

### ISA 定义的中断

所有异常类型的陷阱在 RISC-V 上都非常容易处理，因为我们基本上只是将控制权直接交还给相关的通用 Linux 例程。然而，中断是另一回事：据我所知，对于我们在 RISC-V 上有的第一级中断复用，没有任何通用的基础设施，所以至少目前我们有自己的机制。我们现在的机制有点混乱，所以虽然我将在这里描述正在发生的事情，但这一切都在变化中，当你读到这篇博客文章时可能已经改变了。

描述为什么这很混乱的最好方法是，以定时器中断为例走一遍流程：

*   SEE 实现（监督者执行环境，即底层的固件）确定一个定时器中断已经发生，并进入监督者的陷阱处理程序，`scause` 被相应设置，在 Linux 中这个处理程序是 `handle_exception`。
*   Linux 通过查看 `scause` 中的一个位来确定这是一个中断，然后调用 `do_IRQ` 来处理中断。
*   `do_IRQ` 调用 RISC-V 中断控制器驱动程序的中断处理函数 `riscv_intc_irq`。这是第一点混乱之处：我们的核心架构移植通过一个 RISC-V 特定的 API 与我们的一个驱动程序绑定在一起，这通常是个坏主意。
*   `riscv_intc_irq` 确定这是一个定时器中断，然后通过 `riscv_timer_interrupt` 回调到核心 RISC-V 架构移植来处理定时器中断。这是另一个 RISC-V 特定的 API，但这个没那么吓人，因为依赖关系是从驱动到架构移植，这很难避免。
*   `riscv_timer_interrupt` 查看 RISC-V 定时器驱动程序的 percpu 数据结构以找到相关的 `struct clock_event_device`，然后可以调用它来处理定时器中断。这是我们的架构移植到驱动程序的又一个依赖，这也是个坏主意。

将我们的架构移植与我们的驱动程序交织在一起通常是个坏主意。虽然我们过去曾将我们的驱动程序更紧密地融入到我们的架构移植中，并且正在清理这个过程，但要使这一切都变得合理，还有更多的工作要做。

### PLIC

与我们第一级中断处理流程的工作方式形成鲜明对比的是，PLIC 驱动程序有一个相当干净的中断处理流程。注册 PLIC 驱动程序完全通过标准的设备树机制处理，并且处理中断不触及任何核心 RISC-V 代码。

> 译者注：PLIC (Platform-Level Interrupt Controller) 是 RISC-V 平台级中断控制器，用于管理来自外部设备的中断请求，并将它们路由到 harts。

`plic_init`，即 PLIC 驱动程序的初始化函数，挂接到 Linux 的 `irqchip` 设备树基础设施中，因此当设备树中存在 PLIC 时，它就可以被调用。PLIC 的中断映射也由设备树指定，因此为了注册各种中断处理程序，PLIC 驱动程序需要做的就是挂接到 Linux 的通用 IRQ 注册子系统中，以将一个 PLIC 中断链接到它应该触发的任何设备驱动程序。

实际的 PLIC 中断处理流程相当简单：由于 PLIC 是为现代多核中断处理流程设计的，我们不需要跳过很多圈圈来处理一个中断。一般的 PLIC 中断处理流程如下：

*   一个给定的（hart 上下文，中断 ID）对被启用。在当前的 Linux 驱动中，我们暂时只支持全局启用中断，但 PLIC 被设计为在未来某个时候允许按 hart 进行中断路由。
*   虽然 PLIC 硬件允许优先级和阈值，但我们目前在 Linux 驱动中不支持这些。因此，我们只是简单地配置阈值以允许所有中断进入，并根据请求启用或禁用它们。
*   一旦中断被启用并且设备最终触发了一个中断，PLIC 将选择一个 hart 上下文（其中触发的中断被启用并且超过该上下文的阈值）来中断，通过拉高本地中断控制器的外部中断线，从而导致控制进入内核的陷阱入口点，并最终过滤到 PLIC 驱动程序注册的中断处理例程。
*   正如在许多中断处理流程中常见的那样，PLIC 引发的外部中断仅仅意味着“可能有任意数量的工作要做，你应该现在去检查”。在处理 PLIC 中断时，这会触发一个有三个阶段的中断处理循环。
    + 首先，驱动程序通过 `plic_claim` 函数轮询该 hart 的中断声明寄存器。这个内存映射读操作作为一个同步点：它既通知软件有一个待处理的中断应该被处理（保留了哨兵值 0 来终止处理循环），也允许 PLIC 硬件确保一个中断只被单个 hart 上下文处理。
    + 然后，PLIC 驱动程序将 PLIC 的 IRQ 编号映射到相应的 Linux 中断处理程序（通过 Linux 的通用 IRQ 子系统），然后允许 Linux 处理该中断。
    + 为了完成一个中断，PLIC 驱动程序然后通过 `plic_complete` 函数通知 PLIC 硬件它已经处理完中断，该函数将中断 ID 写回声明寄存器。这会在 PLIC 中重新启用该中断，允许它再次被处理。
*   只要 `plic_claim` 返回待处理的中断，PLIC 驱动程序就会继续处理中断，这允许多个中断被处理而无需引入额外的上下文切换。
*   一旦 PLIC 驱动程序确定没有更多待处理的中断，它会通知通用 IRQ 处理子系统它暂时处理完中断，然后返回。

目前状态下的 PLIC 驱动程序状况相当不错，但还有一些小的清理工作可以做：

*   PLIC 假设中断源 ID 可以在 Linux 中全局注册，这可能会与系统中的其他中断控制器冲突。目前我们只有本地中断控制器，它不与通用 IRQ 处理子系统注册，所以这可能会改变。
*   PLIC 硬件支持 AMO，并被设计为有一个高效的软件实现，使用 AMO 来启用和禁用中断。由于我们目前无法在 RISC-V 系统上查询 PMA（物理内存属性）以确保 Linux 确实可以使用 AMO 与 PLIC 通信，我们现在在 PLIC 驱动中没有利用这一点。
*   PLIC 使用简单的 IRQ 处理流程，但它可能更适合 FastEOI 流程。这将允许我们通过将中断轮询循环从 RISC-V 特定代码中提升出来并使用通用版本来清理 PLIC 驱动程序的实现。

希望在 PLIC 驱动程序进入上游时，我们能解决其中一些问题，但由于许多问题都相当次要，我们可能需要一些时间来清理所有东西。

### 保存和恢复扩展状态

RISC-V 被设计成一个可扩展的 ISA，因此它有多个扩展。Linux 对这些扩展基本上是无感的：它强制要求 A 扩展，并且可能会在有 M 和 C 扩展的系统上运行，但由于这些扩展没有任何额外的状态，这些都是容易处理的事情。

目前 RISC-V 标准定义的唯一增加了用户可见状态的扩展是 F 和 D 扩展，它们增加了浮点寄存器。RISC-V 浮点扩展遵循标准模式：Linux 能够将寄存器状态标记为“访问时陷阱”（trap on access），这样它就可以惰性地保存和恢复 F 寄存器状态。这有一个主要优点：由于在没有 F 寄存器的系统上，F 寄存器状态永远不会是脏的（dirty），我们不需要在 Linux 内核中有任何显式代码来区分有或没有 F 扩展的系统。

### 返回用户空间

在 RISC-V 系统上返回用户空间相当直接：由于硬件在上下文切换时做的事情很少，我们只需反转进入内核时发生的一切，并发出一条 `sret` 指令回到用户空间。

在返回用户空间之前，我们需要做一些 Linux 特定的事情：

*   如果我们是从一个系统调用返回，我们需要检查系统调用跟踪是否已启用。这允许 ptrace 接口工作，这是像 GDB 和 strace 这样的程序使用的内核接口。
*   在以任何方式返回用户空间之前，我们检查是否在进入用户空间之前有其他工作要做，必要时调用一些信号处理程序或调度器。
*   `tp` 与 `sscratch` 交换，这样内核在再次被进入时可以再次找到其内部数据结构。
*   恢复用户空间的整数寄存器状态副本，并跳回用户空间。

## 信号处理

由于信号处理是 Linux 中较为复杂的方面之一，我们尽量不偏离标准的 asm-generic 机制。RISC-V 信号处理基础设施主要围绕着向用户空间传递一个 `struct sigcontext`，其中包含异常发生时用户的架构状态（由内核保存），给由用户空间注册的信号处理函数。

用户空间的信号处理函数具有与常规函数相同的 ABI。这使得进入信号处理程序变得容易：我们只需修改 `sepc` 以包含信号处理程序的地址，然后通过正常机制返回用户空间。信号处理程序被期望在完成时调用 `sigreturn`，因此为了维持常规函数 ABI，我们将返回链接设置为一个基于 VDSO 的蹦床函数（trampoline function）来做这件事。

> 译者注：VDSO (Virtual Dynamic Shared Object) 是内核映射到每个用户进程地址空间的一小块内存，包含一些可以在用户态直接执行的内核代码，如 `sigreturn` 蹦床，以避免陷入内核的开销。

由于 `struct sigcontext` 包含了在捕获到要处理的信号时所有用户可见的、架构定义的状态的副本，这需要是可扩展的，以允许未来的 RISC-V ISA 扩展能够与需要查询这些上下文的应用程序干净地集成。为了支持这些应用程序，我们为 `struct sigcontext` 定义了一个可扩展的格式，允许这些未来的扩展对用户空间可见。我们目前只支持 F 和 D 扩展，但我们在设计时已经考虑到了最终的 V 扩展。
