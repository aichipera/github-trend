# [一边天堂，一边地狱：RISC-V 在开源世界的高光与“垃圾代码”时刻](20250812-riscv_debian_linus_drama.mp3)

本周，对于正在崛起的开放指令集架构 RISC-V 而言，是充满戏剧性的一周。两则重磅新闻从不同维度，将其推到了科技舞台的聚光灯下，完美诠释了何为“冰火两重天”。

## 载入史册：Debian 13 “Trixie” 正式拥抱 RISC-V

首先是令人振奋的“高光时刻”。经过长达两年的开发，备受期待的 Linux 发行版 [Debian 13 "Trixie" 正式发布](https://www.hackster.io/news/debial-13-trixie-lands-with-risc-v-support-but-prepares-to-drop-a-few-other-architectures-709718df4bc5)。这次更新的最大亮点之一，便是首次将 64 位 RISC-V（riscv64）纳为官方支持的体系架构。

这意味着，用户可以在 RISC-V 硬件上轻松地安装和运行这一以稳定著称的操作系统，享受完整的软件包生态和社区支持。这无疑是 RISC-V 发展史上的一个重要里程碑，标志着它已从“爱好者”和“实验性”的标签下走出，正式被主流、稳定的开源社区所接纳，为其在服务器、桌面及嵌入式领域的应用铺平了道路。

有趣的是，在拥抱新架构的同时，Debian 13 也果断地向历史告别，宣布不再将 i386（32位x86）作为常规架构支持，并计划让 armel 架构在这一版本后“退役”。这一“新陈代谢”的举动，更加凸显了 Debian 对 RISC-V 未来潜力的看好与战略布局。

## “垃圾代码，让世界变得更糟！”：Linus Torvalds 的经典咆哮

然而，正当 RISC-V 社区沉浸在喜悦中时，另一则新闻带来了刺骨的“寒流”。Linux 的创造者、以“暴脾气”和技术洁癖闻名的 Linus Torvalds，在内核邮件列表中，对一份提交给 Linux 6.17 的 RISC-V 相关补丁发起了猛烈抨击。

这份由 Meta 工程师 Palmer Dabbelt 提交的补丁，在合并窗口即将关闭的“最后一刻”才姗姗来迟。但这并非问题的关键，Linus 的怒火主要集中在代码质量上。他直言不讳地称其为[“垃圾”（garbage）](https://www.theregister.com/2025/08/11/torvalds_blasts_tardy_kernel_dev/)，并认为其中一个辅助函数“让世界变得更糟”（makes the world actively a worse place to live）。

Linus 愤怒的核心在于，这些质量低劣、逻辑混乱的代码被添加到了通用的内核头文件中，而非限定在 RISC-V 的特定目录下。这意味着它可能会污染整个内核代码库，影响其他架构的开发者，埋下潜在的风险。他用近乎咆哮的语气警告道：“你被警告了：别再有迟到的拉取请求，也别再有 RISC-V 目录之外的垃圾。”

最终，这份补丁被无情拒绝，相关功能至少要等到 Linux 6.18 版本，前提是“尽早提交，并且不含垃圾”。

## 从“新宠”到“学徒”：RISC-V 成熟之路的必要试炼

将这两件事并置，我们看到了一幅生动的图景：RISC-V 正处在一个关键的转折点。它不再是象牙塔里的学术项目，而是已经踏入真实世界的“战场”。

Debian 的接纳，代表了应用生态和广大开发者社区的认可与欢迎。这是它作为“新宠”获得的光环。而 Linus 的“咆哮”，则是它作为“学徒”在进入最高技术殿堂——Linux 内核时，所必须面对的严苛考验。

Linus 的角色，正是顶级开源项目的“质量守门员”。他的严苛甚至“不近人情”，确保了 Linux 内核三十年来的稳定与卓越。这次事件并非针对 RISC-V 本身，而是对所有试图向内核贡献代码的开发者的一次纪律重申：**在这里，截止日期和代码质量是神圣不可侵犯的铁律。**

对于 RISC-V 而言，这盆“冷水”或许比一味的赞扬更有价值。它提醒所有参与者，要想在核心基础设施中站稳脚跟，光有开放的理念和社区的热情是远远不够的，还必须遵循最严格的工程规范和质量标准。

这周的冰与火，恰是 RISC-V 成熟之路上一次宝贵的公开课。它既收获了进入主流的门票，也领教了顶级赛场的规则。前路漫漫，但方向已然清晰。
